<span data-ttu-id="e6f0b-109">Les tâches sont des constructions utilisées pour implémenter ce que l’on appelle le [modèle de promesses de concurrence](https://en.wikipedia.org/wiki/Futures_and_promises).</span><span class="sxs-lookup"><span data-stu-id="e6f0b-109">Tasks are constructs used to implement what is known as the [Promise Model of Concurrency](https://en.wikipedia.org/wiki/Futures_and_promises).</span></span>  <span data-ttu-id="e6f0b-110">En bref, elles vous offrent la « promesse » que le travail sera terminé à un moment ultérieur, ce qui vous permet de coordonner la promesse et une nouvelle API.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-110">In short, they offer you a "promise" that work will be completed at a later point, letting you coordinate with the promise with a clean API.</span></span>

<span data-ttu-id="e6f0b-116">Les tâches exposent un protocole d’API pour surveiller et attendre la valeur de résultat d’une tâche et y accéder (dans le cas de `Task<T>`).</span><span class="sxs-lookup"><span data-stu-id="e6f0b-116">Tasks expose an API protocol for monitoring, waiting upon and accessing the result value (in the case of `Task<T>`) of a task.</span></span> <span data-ttu-id="e6f0b-117">L’intégration au langage, avec le mot clé `await`, fournit une abstraction de niveau supérieur pour l’utilisation des tâches.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-117">Language integration, with the `await` keyword, provides a higher-level abstraction for using tasks.</span></span>

## <a name="deeper-dive-into-tasks-for-an-io-bound-operation"></a><span data-ttu-id="e6f0b-124">Approfondissement : Tâches pour une opération utilisant des E/S</span><span class="sxs-lookup"><span data-stu-id="e6f0b-124">Deeper Dive into Tasks for an I/O-Bound Operation</span></span>

<span data-ttu-id="e6f0b-125">La section suivante décrit en détail toutes les étapes d’un appel d’E/S asynchrone standard.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-125">The following section describes a 10,000 foot view of what happens with a typical async I/O call.</span></span> <span data-ttu-id="e6f0b-126">Commençons par deux exemples.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-126">Let's start with a couple examples.</span></span>

<span data-ttu-id="e6f0b-127">Le premier exemple appelle une méthode async et retourne une tâche active qui n’est pas encore terminée.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-127">The first example calls an async method and returns an active task, likely yet to complete.</span></span>

```csharp
public Task<string> GetHtmlAsync()
{
    // Execution is synchronous here
    var client = new HttpClient();
    
    return client.GetStringAsync("http://www.dotnetfoundation.org");
}
```

<span data-ttu-id="e6f0b-128">Le deuxième exemple ajoute l’utilisation des mots clés `async` et `await` pour agir sur la tâche.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-128">The second example adds the use of the `async` and `await` keywords to operate on the task.</span></span>

```csharp
public async Task<string> GetFirstCharactersCountAsync(string url, int count)
{
    // Execution is synchronous here
    var client = new HttpClient();
    
    // Execution of GetFirstCharactersCountAsync() is yielded to the caller here
    // GetStringAsync returns a Task<string>, which is *awaited*
    var page = await client.GetStringAsync("http://www.dotnetfoundation.org");
    
    // Execution resumes when the client.GetStringAsync task completes,
    // becoming synchronous again.
    
    if (count > page.Length)
    {
        return page;
    }
    else
    {
        return page.Substring(0, count);
    }
}
```

<span data-ttu-id="e6f0b-129">L’appel de `GetStringAsync()` s’effectue par le biais de bibliothèques .NET de niveau inférieur (peut-être en appelant d’autres méthodes async) jusqu’à ce qu’il atteigne un appel interop P/Invoke dans une bibliothèque de réseau native.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-129">The call to `GetStringAsync()` calls through lower-level .NET libraries (perhaps calling other async methods) until it reaches a P/Invoke interop call into a native networking library.</span></span> <span data-ttu-id="e6f0b-130">La bibliothèque native peut ensuite effectuer un appel de l’API système (tel que `write()` pour un socket sur Linux).</span><span class="sxs-lookup"><span data-stu-id="e6f0b-130">The native library may subsequently call into a System API call (such as `write()` to a socket on Linux).</span></span> <span data-ttu-id="e6f0b-131">Un objet de tâche est créé dans la limite native/managée, éventuellement à l’aide de [TaskCompletionSource](xref:System.Threading.Tasks.TaskCompletionSource%601.SetResult(%600)).</span><span class="sxs-lookup"><span data-stu-id="e6f0b-131">A task object will be created at the native/managed boundary, possibly using [TaskCompletionSource](xref:System.Threading.Tasks.TaskCompletionSource%601.SetResult(%600)).</span></span> <span data-ttu-id="e6f0b-132">L’objet de tâche est transmis à travers les couches, éventuellement traité ou directement retourné, ou retourné à l’appelant initial.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-132">The task object will be passed up through the layers, possibly operated on or directly returned, eventually returned to the initial caller.</span></span> 

<span data-ttu-id="e6f0b-133">Dans le deuxième exemple ci-dessus, un objet `Task<T>` est retourné par `GetStringAsync`.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-133">In the second example above, a `Task<T>` object will be returned from `GetStringAsync`.</span></span> <span data-ttu-id="e6f0b-134">L’utilisation du mot clé `await` indique à la méthode de retourner un objet de tâche nouvellement créé.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-134">The use of the `await` keyword causes the method to return a newly created task object.</span></span> <span data-ttu-id="e6f0b-135">Le contrôle retourne à l’appelant à partir de cet emplacement dans la méthode `GetFirstCharactersCountAsync`.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-135">Control returns to the caller from this location in the `GetFirstCharactersCountAsync` method.</span></span> <span data-ttu-id="e6f0b-136">Les méthodes et propriétés de l’objet [Task&lt;T&gt;](xref:System.Threading.Tasks.Task%601) permettent aux appelants de surveiller la progression de la tâche, qui se termine quand le code restant dans GetFirstCharactersCountAsync a été exécuté.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-136">The methods and properties of the [Task&lt;T&gt;](xref:System.Threading.Tasks.Task%601) object enable callers to monitor the progress of the task, which will complete when the remaining code in GetFirstCharactersCountAsync has executed.</span></span>

<span data-ttu-id="e6f0b-137">Après l’appel de l’API système, la demande se trouve dans l’espace du noyau et transite vers le sous-système de réseau du système d’exploitation (comme `/net` dans le noyau Linux).</span><span class="sxs-lookup"><span data-stu-id="e6f0b-137">After the System API call, the request is now in kernel space, making its way to the networking subsystem of the OS (such as `/net` in the Linux Kernel).</span></span>  <span data-ttu-id="e6f0b-138">Ici, le système d’exploitation gère la demande de mise en réseau de manière *asynchrone*.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-138">Here the OS will handle the networking request *asynchronously*.</span></span>  <span data-ttu-id="e6f0b-139">Les détails peuvent être différents selon le système d’exploitation utilisé (l’appel du pilote de périphérique peut être planifié comme un signal envoyé au runtime, ou il peut être effectué et *ensuite* un signal est renvoyé), mais finalement le runtime est informé que la demande de mise en réseau est en cours.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-139">Details may be different depending on the OS used (the device driver call may be scheduled as a signal sent back to the runtime, or a device driver call may be made and *then* a signal sent back), but eventually the runtime will be informed that the networking request is in progress.</span></span>  <span data-ttu-id="e6f0b-140">À ce stade, le travail du pilote de périphérique est planifié, en cours ou déjà terminé (la demande est déjà « sur le réseau »), mais parce que tout se passe de manière asynchrone, le pilote de périphérique est en mesure de gérer immédiatement autre chose !</span><span class="sxs-lookup"><span data-stu-id="e6f0b-140">At this time, the work for the device driver will either be scheduled, in-progress, or already finished (the request is already out "over the wire") - <span data-ttu-id="e6f0b-141">Par exemple, dans Windows, un thread de système d’exploitation effectue un appel au pilote de périphérique réseau et lui demande d’effectuer l’opération de mise en réseau via un paquet de requêtes d’interruption qui représente l’opération.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-141">For example, in Windows an OS thread makes a call to the network device driver and asks it to perform the networking operation via an Interrupt Request Packet (IRP) which represents the operation.</span></span>  <span data-ttu-id="e6f0b-142">Le pilote de périphérique reçoit le paquet de requêtes d’interruption, effectue l’appel au réseau, marque le paquet comme étant « en attente » et le renvoie au système d’exploitation.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-142">The device driver receives the IRP, makes the call to the network, marks the IRP as "pending", and returns back to the OS.</span></span>  <span data-ttu-id="e6f0b-143">Le thread du système d’exploitation sait maintenant que le paquet de requêtes d’interruption est « en attente », il n’a donc rien d’autre à faire pour ce travail et « revient » pour pouvoir être utilisé pour une autre opération.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-143">Because the OS thread now knows that the IRP is "pending", it doesn't have any more work to do for this job and "returns" back so that it can be used to perform other work.</span></span>

<span data-ttu-id="e6f0b-144">Quand la demande est satisfaite et que les données reviennent à travers le pilote de périphérique, il avertit le processeur que de nouvelles données sont reçues via une interruption.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-144">When the request is fulfilled and data comes back through the device driver, it notifies the CPU of new data received via an interrupt.</span></span>  <span data-ttu-id="e6f0b-145">La façon dont cette interruption est gérée varie selon le système d’exploitation, mais les données sont ensuite transmises au système d’exploitation jusqu’à ce que se produise un appel d’interopérabilité système (par exemple, dans Linux, un gestionnaire d’interruptions planifie la moitié inférieure de l’IRQ pour qu’elle transmette les données via le système d’exploitation de façon asynchrone).</span><span class="sxs-lookup"><span data-stu-id="e6f0b-145">How this interrupt gets handled will vary depending on the OS, but eventually the data will be passed through the OS until it reaches a system interop call (for example, in Linux an interrupt handler will schedule the bottom half of the IRQ to pass the data up through the OS asynchronously).</span></span>  <span data-ttu-id="e6f0b-146">Notez que cela se produit *également* de façon asynchrone !</span><span class="sxs-lookup"><span data-stu-id="e6f0b-146">Note that this *also* happens asynchronously!</span></span>  <span data-ttu-id="e6f0b-147">Le résultat est placé en file d’attente jusqu’à ce que le prochain thread disponible soit en mesure d’exécuter la méthode async et de « désencapsuler » le résultat de la tâche effectuée.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-147">The result is queued up until the next available thread is able execute the async method and "unwrap" the result of the completed task.</span></span>

<span data-ttu-id="e6f0b-148">Tout au long de ce processus, un élément clé à retenir est qu’**aucun thread n’est dédié à l’exécution de la tâche**.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-148">Throughout this entire process, a key takeaway is that **no thread is dedicated to running the task**.</span></span>  <span data-ttu-id="e6f0b-149">Bien que le travail soit exécuté dans un contexte (c’est-à-dire que le système d’exploitation doit passer des données à un pilote de périphérique et répondre à une interruption), aucun thread n’est destiné à *attendre* le retour des données de la demande.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-149">Although work is executed in some context (that is, the OS does have to pass data to a device driver and respond to an interrupt), there is no thread dedicated to *waiting* for data from the request to come back.</span></span>  <span data-ttu-id="e6f0b-150">Cela permet au système de gérer une plus grande quantité de travail au lieu d’attendre la fin des appels d’E/S.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-150">This allows the system to handle a much larger volume of work rather than waiting for some I/O call to finish.</span></span>

<span data-ttu-id="e6f0b-158">Ce modèle fonctionne correctement avec la charge de travail d’un scénario de serveur classique.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-158">This model works well with a typical server scenario workload.</span></span>  <span data-ttu-id="e6f0b-159">Comme aucun thread n’est destiné à s’interrompre sur les tâches non terminées, le pool de threads serveur peut traiter un plus grand nombre de demandes web.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-159">Because there are no threads dedicated to blocking on unfinished tasks, the server threadpool can service a much higher volume of web requests.</span></span>

<span data-ttu-id="e6f0b-160">Prenons deux serveurs : l’un d’eux exécute du code asynchrone et l’autre pas.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-160">Consider two servers: one that runs async code, and one that does not.</span></span>  <span data-ttu-id="e6f0b-161">Pour les besoins de cet exemple, chaque serveur n’a que 5 threads disponibles pour traiter les demandes.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-161">For the purpose of this example, each server only has 5 threads available to service requests.</span></span>  <span data-ttu-id="e6f0b-162">Notez que ces chiffres sont intentionnellement petits et servent uniquement dans un contexte de démonstration.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-162">Note that these numbers are imaginarily small and serve only in a demonstrative context.</span></span>

<span data-ttu-id="e6f0b-163">Supposons que les deux serveurs reçoivent 6 demandes simultanées.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-163">Assume both servers receive 6 concurrent requests.</span></span> <span data-ttu-id="e6f0b-164">Chaque demande effectue une opération d’E/S.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-164">Each request performs an I/O operation.</span></span>  <span data-ttu-id="e6f0b-165">Le serveur *sans* code asynchrone doit placer en file d’attente la 6ème demande jusqu’à ce que l’un des 5 threads ait terminé le travail utilisant des E/S et écrit une réponse.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-165">The server *without* async code has to queue up the 6th request until one of the 5 threads have finished the I/O-bound work and written a response.</span></span> <span data-ttu-id="e6f0b-166">Quand la 20ème demande arrive, le serveur commence peut-être à ralentir, car la file d’attente devient trop longue.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-166">At the point that the 20th request comes in, the server might start to slow down, because the queue is getting too long.</span></span>

<span data-ttu-id="e6f0b-167">Le serveur *avec* code asynchrone peut placer en file d’attente la 6ème demande, mais parce qu’il utilise `async` et `await`, chacun de ses threads est libéré quand le travail utilisant des E/S démarre, et non quand il se termine.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-167">The server *with* async code running on it still queues up the 6th request, but because it uses `async` and `await`, each of its threads are freed up when the I/O-bound work starts, rather than when it finishes.</span></span>  <span data-ttu-id="e6f0b-168">Quand la 20ème demande arrive, la file d’attente des demandes entrantes est bien plus petite (ou est totalement vide) et le serveur ne ralentit pas.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-168">By the time the 20th request comes in, the queue for incoming requests will be far smaller (if it has anything in it at all), and the server won't slow down.</span></span>

<span data-ttu-id="e6f0b-169">Bien qu’il s’agisse d’un exemple fictif, il fonctionne de manière très similaire dans le monde réel.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-169">Although this is a contrived example, it works in a very similar fashion in the real world.</span></span>  <span data-ttu-id="e6f0b-170">En réalité, un serveur est capable de gérer un nombre bien plus important de demandes à l’aide de `async` et `await` que s’il dédiait un thread à chaque demande qu’il reçoit.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-170">In fact, you can expect a server to be able to handle an order of magnitude more requests using `async` and `await` than if it were dedicating a thread for each request it receives.</span></span>

### <a name="what-does-this-mean-for-client-scenario"></a><span data-ttu-id="e6f0b-171">Qu’est-ce que cela signifie dans un scénario de client ?</span><span class="sxs-lookup"><span data-stu-id="e6f0b-171">What does this mean for client scenario?</span></span>

<span data-ttu-id="e6f0b-172">Le plus gros avantage de l’utilisation de `async` et `await` pour une application cliente est l’augmentation de la réactivité.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-172">The biggest gain for using `async` and `await` for a client app is an increase in responsiveness.</span></span>  <span data-ttu-id="e6f0b-173">Même si vous pouvez améliorer la réactivité d’une application en gérant manuellement des threads de manière dynamique, c’est une opération coûteuse par rapport à la simple utilisation de `async` et `await`.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-173">Although you can make an app responsive by spawning threads manually, the act of spawning a thread is an expensive operation relative to just using `async` and `await`.</span></span>  <span data-ttu-id="e6f0b-174">Dans le cas particulier d’un jeu mobile, il est essentiel d’affecter aussi peu que possible le thread d’interface utilisateur en ce qui concerne les E/S.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-174">Especially for something like a mobile game, impacting the UI thread as little as possible where I/O is concerned is crucial.</span></span>

<span data-ttu-id="e6f0b-175">Plus important encore, parce que le travail utilisant des E/S ne se sert pratiquement pas du processeur, en dédiant un thread de processeur entier pour effectuer vaguement des tâches utiles, vous utilisez mal vos ressources.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-175">More importantly, because I/O-bound work spends virtually no time on the CPU, dedicating an entire CPU thread to perform barely any useful work would be a poor use of resources.</span></span>

<span data-ttu-id="e6f0b-176">Par ailleurs, la répartition du travail sur le thread d’interface utilisateur (par exemple, la mise à jour d’une interface utilisateur) est très simple avec des méthodes `async` et n’engendre pas de travail supplémentaire (par exemple, l’appel d’un délégué thread-safe).</span><span class="sxs-lookup"><span data-stu-id="e6f0b-176">Additionally, dispatching work to the UI thread (such as updating a UI) is very simple with `async` methods, and does not require extra work (such as calling a thread-safe delegate).</span></span>

## <a name="deeper-dive-into-task-and-taskt-for-a-cpu-bound-operation"></a><span data-ttu-id="e6f0b-177">Approfondissement : Task et Task<T> pour une opération utilisant le processeur</span><span class="sxs-lookup"><span data-stu-id="e6f0b-177">Deeper Dive into Task and Task<T> for a CPU-Bound Operation</span></span>

<span data-ttu-id="e6f0b-178">Le code `async` utilisant le processeur est un peu différent du code `async` utilisant des E/S.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-178">CPU-bound `async` code is a bit different than I/O-bound `async` code.</span></span>  <span data-ttu-id="e6f0b-179">Comme le travail est effectué sur le processeur, il n’est pas possible de dédier un thread au calcul.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-179">Because the work is done on the CPU, there's no way to get around dedicating a thread to the computation.</span></span>  <span data-ttu-id="e6f0b-180">L’utilisation de `async` et `await` est un moyen d’interagir avec un thread en arrière-plan et de faire en sorte que l’appelant de la méthode async reste réactif.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-180">The use of `async` and `await` provides you with a clean way to interact with a background thread and keep the caller of the async method responsive.</span></span>  <span data-ttu-id="e6f0b-181">Notez que cela ne protège en rien les données partagées.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-181">Note that this does not provide any protection for shared data.</span></span>  <span data-ttu-id="e6f0b-182">Si vous utilisez des données partagées, vous devez quand même appliquer une stratégie de synchronisation appropriée.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-182">If you are using shared data, you will still need to apply an appropriate synchronization strategy.</span></span>

<span data-ttu-id="e6f0b-183">Voici une vue générale d’un appel asynchrone utilisant le processeur :</span><span class="sxs-lookup"><span data-stu-id="e6f0b-183">Here's a 10,000 foot view of a CPU-bound async call:</span></span>

```csharp
public async Task<int> CalculateResult(InputData data)
{
    // This queues up the work on the threadpool.
    var expensiveResultTask = Task.Run(() => DoExpensiveCalculation(data));
    
    // Note that at this point, you can do some other work concurrently,
    // as CalculateResult() is still executing!
    
    // Execution of CalculateResult is yielded here!
    var result = await expensiveResultTask;
    
    return result;
}
```

<span data-ttu-id="e6f0b-184">`CalculateResult()` s’exécute sur le thread sur lequel il a été appelé.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-184">`CalculateResult()` executes on the thread it was called on.</span></span>  <span data-ttu-id="e6f0b-185">Quand il appelle `Task.Run`, il place en file d’attente l’opération coûteuse qui utilise le processeur, `DoExpensiveCalculation()`, sur le pool de threads et reçoit un handle `Task<int>`.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-185">When it calls `Task.Run`, it queues the expensive CPU-bound operation, `DoExpensiveCalculation()`, on the thread pool and receives a `Task<int>` handle.</span></span>  <span data-ttu-id="e6f0b-186">`DoExpensiveCalculation()` est finalement exécuté simultanément sur le prochain thread disponible, probablement sur un autre cœur d’UC.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-186">`DoExpensiveCalculation()` is eventually run concurrently on the next available thread, likely on another CPU core.</span></span>  <span data-ttu-id="e6f0b-187">Il est possible d’effectuer des tâches simultanées quand `DoExpensiveCalculation()` est occupé sur un autre thread, car le thread qui a appelé `CalculateResult()` est encore en cours d’exécution.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-187">It's possible to do concurrent work while `DoExpensiveCalculation()` is busy on another thread, because the thread which called `CalculateResult()` is still executing.</span></span>

<span data-ttu-id="e6f0b-188">Une fois que `await` a été trouvé, l’exécution de `CalculateResult()` est cédée à son appelant, ce qui permet d’effectuer d’autres tâches avec le thread actuel pendant que `DoExpensiveCalculation()` produit un résultat.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-188">Once `await` is encountered, the execution of `CalculateResult()` is yielded to its caller, allowing other work to be done with the current thread while `DoExpensiveCalculation()` is churning out a result.</span></span>  <span data-ttu-id="e6f0b-189">Une fois cette opération terminée, le résultat est placé en file d’attente pour s’exécuter sur le thread principal.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-189">Once it has finished, the result is queued up to run on the main thread.</span></span>  <span data-ttu-id="e6f0b-190">Finalement, le thread principal retourne à l’exécution de `CalculateResult()`, à partir duquel il obtient le résultat de `DoExpensiveCalculation()`.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-190">Eventually, the main thread will return to executing `CalculateResult()`, at which point it will have the result of `DoExpensiveCalculation()`.</span></span>

### <a name="why-does-async-help-here"></a><span data-ttu-id="e6f0b-191">Pourquoi async est-il utile ici ?</span><span class="sxs-lookup"><span data-stu-id="e6f0b-191">Why does async help here?</span></span>

<span data-ttu-id="e6f0b-192">`async` et `await` sont la meilleure approche pour gérer les travaux utilisant l’UC quand vous avez besoin de réactivité.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-192">`async` and `await` are the best practice managing CPU-bound work when you need responsiveness.</span></span> <span data-ttu-id="e6f0b-193">Il existe plusieurs modèles d’utilisation d’async avec des tâches utilisant le processeur.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-193">There are multiple patterns for using async with CPU-bound work.</span></span> <span data-ttu-id="e6f0b-194">Notez que l’utilisation d’async représente un coût, même s’il est faible, et qu’elle n’est donc pas recommandée pour les boucles serrées.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-194">It's important to note that there is a small cost to using async and it's not recommended for tight loops.</span></span>  <span data-ttu-id="e6f0b-195">C’est à vous de déterminer la façon dont vous écrivez votre code autour de cette nouvelle fonctionnalité.</span><span class="sxs-lookup"><span data-stu-id="e6f0b-195">It's up to you to determine how you write your code around this new capability.</span></span>
